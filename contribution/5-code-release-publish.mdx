# 🚂 Build, Release & Publish

When developing Crossplane Providers, we propose one process across all projects to ensure high quality and reliability.

This guide walks you through the steps to **build, release, and publish** your changes effectively.

## Code

### Build

We begin by bringing our code changes into a shippable state by following our Definition of Done.

**We encourage all contributors to follow and share these practices.**

### Definition of Done

Any resource or feature we ship should fulfill the following criteria and reflect in their versioning. Requirements for each version build on top of each other.

### Build release candidate

Once your code changes are ready to be shipped, the next step is to create a release candidate.

To create a release candidate, please follow the steps below.


1. Open the Github reposentory of the Project you want to create a release candidate for
2. Go to the Actions tab
![Open Github Actions](/img/contribution_open_github_actions.png)
3. Select `Publish Release Candidate` workflow on the left (could also be named differently depending on the project for example `.github/workflows/release-candidate.yaml`)
<div style={{maxWidth: "220px", margin: "0 auto"}}>
![Select `Publish Release Candidate`](/img/contribution_open_publish_release_candidate.png)
</div>
4. Select `Run workflow` and the branch you want to create the release candidate from and click on `Run workflow`
![Select `Publish Release Candidate`](/img/contribution_search_for_branch.png)
5. Go back to the Github Project start page and select on the right site in the `Packages` section the matching package (e.g. `crossplane-provider-btp`)
<div style={{maxWidth: "220px", margin: "0 auto"}}>
![Select `crossplane-provider-<name>`](/img/contribution_select_package.png)
</div>
6. Select the newly created image that matches your commit hash
![Select your image](/img/contribution_select_your_image.png)
7. Share the image URL with for example the issuer of the Bug to validate your fix (e.g. `ghcr.io/sap/crossplane-provider-btp/crossplane/provider-btp::sha-<commit-hash>`)

### Validate Build

To validate an build, please follow the steps below to use the release candidate image created for your fix.

1. Create a KinD cluster locally or use a cluster in your development environment.
2. Set up the needed artifactory pull secret in the cluster to be able to pull the release candidate image.
3. Install the release candidate provider by the following yaml. Don't forget to replace the image URL with the one created for your fix.

```yaml {6}
apiVersion: pkg.crossplane.io/v1
kind: Provider
metadata:
    name: btp-provider
spec:
    package: <your-release-candidate-image-url>
    packagePullSecrets:
        - name: artifactory-readonly-docker
```

## Publish a release

Once the release candidate has been validated, we want to publish it, for some fixes/features it makes sense to directly publish a new release, for other ones it makes sense to wait and combine it with other changes in a new release.

If your feature/fix is already on the `main` branch it will be part of the next release, if not please start the process to merge your code changes into `main`.
For important bug fixes and features let's start a discussion if it makes sense to directly publish a new release or wait for the next one.
